

# 03_技术架构设计

# 1. 架构目标

本文档定义 Video Creator Tool 的技术实现架构。

架构目标：

- 支撑剧本到视频生产的完整流程
- 支持人工审核节点
- 支持AI能力调用
- 支持未来扩展多视频模型
- 保证生产过程可追踪

系统采用：

前端工作台 + 后端业务服务 + AI服务层 + 异步任务系统 + 数据存储

架构模式。

# 2. 总体架构

```text

                 用户
                  |
                  ↓
          Web前端工作台
                  |
                  ↓
            API Backend
                  |
      ┌───────────┼───────────┐
      ↓           ↓           ↓
 业务服务层    AI服务层    任务调度层
      |           |           |
      ↓           ↓           ↓
 PostgreSQL   OpenAI      Redis Queue
              Seedance        |
              Kling           ↓
                         Worker
                              |
                              ↓
                         Video API
                              |
                              ↓
                         文件存储
```

# 3. 技术分层设计

# 3.1 前端层 Frontend

## 职责

负责用户交互和生产流程展示。

主要页面：

- Dashboard
- Project
- Script输入
- Storyboard审核
- Production Decision
- AI生产准备
- 视频审核

## 技术建议

第一版：

- React
- TypeScript
- Tailwind CSS

原因：

适合快速构建复杂工作台界面。

# 3.2 后端业务层 Backend

## 职责

负责：

- 用户请求处理
- 业务流程控制
- 状态流转
- 数据管理
- AI调用编排

## 技术建议

Python FastAPI。

原因：

- AI生态成熟
- 方便调用模型API
- 开发效率高

# 3.3 数据层 Database

## PostgreSQL

保存结构化业务数据：

- Project
- Script
- Storyboard
- Shot
- Review
- Production Task
- Generation Result

## Redis

用途：

- 任务队列
- 状态缓存
- 异步任务管理

# 3.4 文件存储层 Storage

保存非结构化文件：

- 产品图片
- 首帧图片
- 参考视频
- 生成视频

MVP：

本地文件存储。

后续：

对象存储（S3/OSS/MinIO）。

# 4. 核心服务模块

# 4.1 Script Service

负责：

剧本管理。

功能：

- 创建剧本
- 保存版本
- 提交分析

调用：

LLM分析服务。

# 4.2 Storyboard Service

负责：

剧本 → 分镜。

输入：

Script

输出：

Storyboard + Shot列表。

# 4.3 Review Service

负责所有人工审核。

包括：

- 分镜审核
- 生产方式审核
- 视频结果审核

核心能力：

记录状态变化和审核历史。

# 4.4 AI Director Service

负责AI镜头生产方案生成。

输入：

Shot

输出：

- Generation Mode
- Prompt
- Negative Prompt
- Camera
- Motion
- Lighting
- Asset Requirement

# 4.5 Asset Service

负责素材管理。

功能：

- 素材上传
- 素材关联Shot
- 素材状态管理

# 4.6 Video Generation Service

负责调用视频模型。

支持：

- Seedance
- Kling
- 即梦

统一接口：

```text
create_generation_task()

query_generation_status()

get_result()
```

# 5. AI调用架构

AI调用不直接阻塞用户请求。

流程：

```text
用户点击生成

↓

创建Production Task

↓

写入数据库

↓

进入任务队列

↓

Worker处理

↓

调用视频API

↓

更新任务状态

↓

通知前端
```

视频生成属于耗时任务，应采用异步任务模式，而不是同步等待请求返回。类似视频生成API通常采用job创建、状态查询、结果回调的模式。 

# 6. 异步任务系统设计

## Queue

Redis Queue。

任务类型：

- storyboard_generate
- prompt_generate
- asset_validate
- video_generate

## Worker

负责执行：

- LLM调用
- Seedance调用
- 文件处理

状态更新：

pending
running
completed
failed

# 7. API设计原则

## 前端API

负责业务交互。

例如：

```text
POST /projects

POST /scripts

POST /storyboards/generate

POST /shots/{id}/review

POST /generation/tasks
```

## AI Provider API

统一封装。

禁止业务层直接调用模型。

例如：

```text
AIProvider
 |
 ├── SeedanceProvider
 ├── KlingProvider
 └── JimengProvider
```

# 8. 状态管理设计

系统核心依赖状态驱动。

主要状态：

Project State

Script State

Shot State

Production Task State

Asset State

Generation Result State

禁止通过页面逻辑直接修改状态。

所有状态变化必须经过Backend Service。

# 10. 架构原则

## 原则1：业务流程和AI能力解耦

模型可以替换，业务流程不变。

## 原则2：所有AI调用任务化

视频生成必须进入任务系统。

## 原则3：人工审核始终存在

AI负责效率，人负责决策。

## 原则4：为未来商业决策系统预留数据接口

保存完整生产数据，为未来分析：

- 哪些镜头有效
- 哪些Prompt有效
- 哪些视频表现好

提供数据基础。
